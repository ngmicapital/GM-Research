Fundamentals
How Mixture of Experts (MoE) Works — and Why It Matters for AI Pricing

In a standard "dense" neural network, every query activates every parameter. If a model has 70 billion parameters, all 70 billion do computational work every time you send a message. That's why larger dense models cost more per token — more parameters, more compute, higher price.

MoE solves this with a routing trick. Instead of one monolithic network, MoE divides the model into N specialised "expert" sub-networks. For any given token, a small "router" network activates only K of those experts (where K is much smaller than N). The router is learned during training — it figures out which experts are best at which types of content.

Mistral Small 4's numbers make sense now. It has 128 experts, 4 active per token. That means only 4/128 = 3% of experts are active at inference time, which corresponds to roughly 6.5B out of 119B total parameters. Your API pricing is proportional to active parameters — hence $0.15/M input, comparable to a 7B dense model, despite the 119B headline number.

The trade-offs aren't free. All 119B parameters still need to live in memory (GPU VRAM), so deployment is expensive even if inference is cheap. MoE models are also harder to quantise to low-bit formats without degrading routing quality, and "expert collapse" — where the router over-specialises and some experts are never used — is a real training problem that teams like Mistral have spent years solving.

Why this matters for the AI landscape: MoE is the architecture behind GPT-4, Grok, Mistral, and increasingly the "mini" variants of frontier models. When comparing pricing and capability, the metric that matters is active parameters at inference, not headline parameter count. A "119B" MoE model costs approximately the same to run as a 7B dense model — which is why the cost-performance frontier has moved so dramatically in the past 12 months.

⚡ When a new "large" model releases with surprisingly low pricing, check if it's MoE. Active parameter count tells you the real inference cost. Total parameter count tells you how much VRAM is needed to run it locally.

During the v2.1.88 release push on March 31, a 59.8 MB JavaScript source map file (intended for internal debugging) was inadvertently included in the @anthropic-ai/claude-code npm package. Fundamentals
How Fine-Tuning Differs from Training from Scratch

When people say a model was "trained on" a dataset, there are two very different things they could mean — and confusing them leads to bad intuitions about what AI can do and how quickly the landscape moves.

Training from scratch means starting with randomly initialised weights and running billions of gradient descent steps across a massive corpus (trillions of tokens for frontier models). This requires enormous compute (hundreds of millions of dollars), months of time, and generates a "base model" with broad, general knowledge. This is what OpenAI, Anthropic, and Google DeepMind do when they build GPT-5.4, Claude Opus 4.6, or Gemini 3.

Fine-tuning starts from a pre-trained base model and continues training on a smaller, curated dataset. The idea is that the base model already "knows" language, reasoning, and world knowledge — fine-tuning steers that knowledge toward a specific domain, style, or task. RLHF (Reinforcement Learning from Human Feedback, used by all major frontier labs) is a form of fine-tuning where the dataset is human preference ratings, not raw text.

Why this matters for open-source: The open-source community rarely trains from scratch (too expensive). Instead, it fine-tunes open-weight base models like LLaMA, Mistral Small 4, or GLM-5. This is why open-source moves fast once a capable base model is released — the community can rapidly specialise it for coding, medical, legal, or trading applications without needing frontier-lab compute budgets.

Why this matters for the AI landscape: Fine-tuning makes powerful AI democratically accessible at low cost. But it also means fine-tuned models inherit all the quirks, biases, and capability ceilings of their base. The quality of the base model sets the ceiling; fine-tuning shapes where within that ceiling the model operates. When a new open-weight frontier model releases, the real action happens in the weeks after — in the community fine-tuning explosion it enables.

⚡ Next time a new open-weight model releases (like Mistral Small 4 last week), the follow-on story 2–4 weeks later is which fine-tuned variants emerge. The coding, reasoning, and domain-specialist derivatives are often more practically useful than the base model itself.

Workflow Update
Claude Code: PermissionDenied Hook + Named Subagents — Supervised Automation Pattern

The PermissionDenied hook (v2.1.88) fires whenever Claude Code's auto mode classifier blocks a tool call. Your hook can inspect the denial reason, decide whether it's safe to override, and return {retry: true} to let the model retry. This creates a programmatic supervision layer that didn't exist before.

The pattern this enables: run Claude in auto mode for speed, but intercept specific denial categories (e.g., allow file writes to your briefings folder but block anything touching system directories) without requiring full manual permission for every step. Combined with the FileChanged, TaskCreated, and CwdChanged hooks from v2.1.83–84, you now have a complete event-driven Claude automation system with fine-grained control.

Named subagents in @ mention typeahead make multi-agent setups easier to manage from the prompt input — no need to remember agent names or configurations when you can @ autocomplete them directly.

Try This Week
Set up a PermissionDenied hook that logs all denied operations to a file — this gives you a visibility layer on what Claude is attempting and being blocked from, which is the first step to calibrating your auto mode permissions correctly for the briefing generation workflow.
⚡ Highest-leverage new primitive this week. If dc's briefing automation ever runs into permission friction (Claude trying to write to unexpected paths, access network resources, etc.), the PermissionDenied hook gives you the observability and control layer to fix it programmatically rather than widening permissions globally.

Research
PRBench — Can AI Reproduce Computational Results from Scientific Papers?

Published: Recent (TMLR 2026 track) · Venue: Transactions on Machine Learning Research

What it is: A benchmark of 30 expert-curated physics tasks where AI agents must reproduce computational results directly from scientific papers — finding the right equations, implementing them, running the computations, and matching the published output.

One-line finding: Current AI agents can follow explicit step-by-step reproduction but struggle when the paper's methods are ambiguous, underspecified, or require domain judgment calls that the paper's authors took for granted.

Why it matters: Scientific reproducibility is a major problem — a significant fraction of published computational results can't be reproduced by other researchers. AI that can systematically audit computational claims would accelerate science. PRBench establishes the baseline for how far we are from that.

⚡ Monitor — not actionable for dc today. The 6-month signal: if AI agents make material PRBench progress, it means they're developing the ability to reason about the gap between what's written and what's implied — a capability that transfers directly to understanding underspecified trading system descriptions and incomplete strategy documentation.
