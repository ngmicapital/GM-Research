Diagrams, Hooks, and Aliases — The Senior Engineer's Playbook for AI-Powered Coding

Source How I AI · Claire Vio feat. John Lindquist
Domain Technology / AI Development Workflows

Executive Summary

  • Mermaid diagrams injected at startup eliminate wasted codebase exploration: By storing your application's architecture as Mermaid diagram markdown files and loading them into Claude Code's system prompt via --append-system-prompt, every session starts with full structural context already in memory. No file reads, no codebase spelunking — answers come faster and edits land correctly because the AI understands dependency chains before you type a word. 04:09–10:30
  • Stop hooks turn Claude Code into a self-correcting quality pipeline: Using the Anthropic Agent SDK, attach a TypeScript stop hook that fires every time Claude finishes a task. It checks for changed files, runs your type checker, feeds errors back to Claude as a blocking message, and — when clean — auto-commits. The loop runs without human input. 27:01–35:15
  • Shell aliases compress your most powerful Claude configurations to one keystroke: ZSH or PowerShell aliases let you launch dangerous mode, switch to Haiku, or fire up a diagram-loaded session with a single letter. The investment is 10 minutes; the payoff is every subsequent session. 19:28–20:54
  • Build every idea immediately as a personal CLI — don't wait for the right project: The moment you have an idea, open a terminal, brain-dump into Claude Code via dictation, and build the smallest possible thing. The constrained UI of the terminal forces focus and eliminates distraction. 21:05–26:43
  • AI's most underused capability is investigation, not generation: Orientation work before touching a legacy codebase (tracing git history, surfacing impact, understanding who wrote what and why) that used to consume a day or two can now be prompted in 30 seconds. The ROI is immediate and unambiguous. 46:33–49:24
  • Plan mode has eliminated most of the "AI drift" problem: When Claude goes off the rails, export the conversation, get a second model to critique it, revert to the previous commit, and start over. But increasingly, plan mode prevents drift from occurring in the first place. 51:57–55:07

🗺️ Mermaid Diagrams as Context Infrastructure — Compressing Your App into Machine-Readable Memory

"Every time an AI starts it has no memory, no idea of what's going on in your application. People try and set up lots of rules but they usually don't include a lot of: how does my application work and how do the pieces fit?" 05:06

What Mermaid Diagrams Are

Mermaid is a standard text format for rendering flowcharts, sequence diagrams, and database schemas inside markdown. A Mermaid diagram compresses an entire authentication flow or database operation into 20–30 lines of structured text. Humans need a visual render to navigate it; AI consumes it directly as dense, semantically rich context. 05:47–06:27

John stores these as markdown files (with embedded Mermaid blocks) in a /memory/ai/diagrams/ directory in every project. Windsurf recently released a similar product called "Codemaps" — AI-annotated hierarchical maps of your codebase — confirming the concept is converging into an industry standard. 04:58–05:05

The --append-system-prompt Command

Running claude --append-system-prompt "$(cat .ai/diagrams/*.md)" loads all diagram files into Claude's system prompt before any user prompt is processed. 07:02–07:52

Result: a query like "explain the authentication flow" returns an immediate, structured answer with zero file reads. Trade-off: significantly more tokens used upfront. John's verdict: "the work that you do, the time that you spend on these tasks is more valuable than that to me." 09:36–09:47

🌐 Consensus Check Windsurf Codemaps (November 2025) and the growing category of "context preloading" tools confirm this as a rapidly converging best practice. Research into LLM context usage consistently shows pre-structured architectural context outperforms in-session retrieval for complex, dependency-heavy tasks.

When to Generate Diagrams

John's timing: after something is working and before a PR is merged. "Once it's working, diagram it out." For greenfield projects: spike first, build until it works, then diagram — not before. 13:59–14:42

Claire runs a GitHub Action that generates diagrams automatically on PR close — a CI-based pattern that scales the workflow team-wide. 13:38–13:57

💡 Key Insight Most developers configure how Claude behaves (via CLAUDE.md, style rules). Fewer configure what Claude knows about their app. Mermaid diagrams fill the architectural knowledge gap that rules files never address. 05:17–05:26
🔗 Connection Diagrams → System Prompt → Faster, more accurate edits. This is the same principle as CLAUDE.md files but applied to architectural structure rather than style preferences. The distinction matters: architecture tells the AI how components relate; rules tell it how to behave. Both are needed. 05:17–05:26

Non-Engineering Use Cases

Security compliance documentation (SOC 2, data flow diagrams for enterprise customers) — historically a painful, expensive manual task, now generatable on demand. Customer-facing technical documentation: generate diagrams from code, summarise for customers, iterate without touching the underlying markdown. 15:22–18:00

🔍 Source Verified ✅ Windsurf Codemaps confirmed. Cognition.ai/blog/codemaps (Nov 2025) — AI-annotated codebase maps "powered by SWE-1.5 and Claude Sonnet 4.5." Includes Mermaid diagram generation, Cascade agent integration, and shareable hierarchical navigation. John's independent approach and Windsurf's product-ised version are architecturally identical in concept.

⚙️ Shell Aliases — Compressing Complex Claude Configurations to One Letter

"Once you have these systems in place, you can just capture them in the smallest shortcuts." 20:20

Every time you manually type a long Claude Code invocation with flags, you're paying a friction tax. Aliases eliminate this. On Mac, ZSH is the default shell; on Windows, PowerShell. Syntax differs; principle is identical. 19:28–19:48

John's Alias Setup (Live Demo)

  • x — claude --dangerously-skip-permissions. Full bypass, no permission prompts. Maximum speed for trusted work.
  • h — Claude using Haiku model. Faster, cheaper, slightly less capable. For quick tasks where Sonnet is overkill.
  • cdi — Claude with the full diagram-loading system prompt pre-injected. One command to start a fully context-loaded session.

Claire's extension: aliases can be project-specific — cc-projectname to pull in only the relevant diagrams for a specific initiative. 20:33–20:52

💡 Key Insight Aliases are the lowest-effort, highest-leverage optimisation in the entire episode. The configuration is once; the saving is every subsequent session. Any engineer not doing this is manually re-entering the same flags every day.

🔧 Stop Hooks — Building a Self-Correcting AI Development Loop

"Once you know part of your workflow always runs, you might as well run it automatically." 35:09

What Stop Hooks Are

A "stop" event fires in Claude Code when Claude has finished its turn and is waiting for user input. A stop hook intercepts that moment and runs a script before control returns. Hooks are configured in settings.local.json (personal) or settings.json (team-shared). The key: team-shared hooks give every engineer the same quality baseline automatically. 28:48–30:13

Hook Architecture (Live Demo)

John's stop hook is a TypeScript script using the Anthropic Agent SDK, which provides strongly-typed HookInput objects containing session ID, working directory, permission mode, and more. 29:38–29:57

  1. Check if any files changed since the last stop event
  2. If files changed → run bun type-check
  3. If TypeScript errors → console.log(JSON.stringify({ decision: "block", reason: errorOutput })) → Claude self-corrects
  4. If no errors → prompt Claude to auto-commit (excluding sensitive files)
  5. If no files changed → exit silently

Demo Result

Claude created a file with a quote syntax error → stop hook fired → Claude received the error report → self-corrected without user intervention → stop hook fired again, found no errors → auto-committed. Two full stop cycles. Zero human input post-task. 33:44–34:47

The console.log Gotcha

The SDK reads your first stdout output as Claude's hook input. Use console.error for any debug logging; use console.log only for the structured JSON response you want Claude to receive. Mixing them causes silent, confusing failures. 32:10–33:08

🔍 Source Verified ✅ Fully confirmed against Anthropic documentation. Claude Code hooks guide (docs.anthropic.com/en/docs/claude-code/hooks-guide) and Agent SDK TypeScript reference (platform.claude.com/docs/en/agent-sdk/hooks) confirm: the stop hook event type, the HookInput schema (sessionId, cwd, permissionMode), and the JSON console.log return contract. The workflow John demonstrates is production-documented.

What Else Belongs in a Stop Hook

Formatting (Prettier, ESLint auto-fix) · Circular dependency detection · File length enforcement · Code complexity analysis · Duplicate code detection ("this code looks like X elsewhere — extract it?") · Any pre-commit or pre-push check that currently runs manually. 36:55–37:52

🎯 Against Consensus Most engineers think of Claude Code as a text-in, text-out interaction tool. John positions it as a programmable quality-gating system. These are fundamentally different mental models with fundamentally different ROI.

Claire's call to engineering leaders: "If you haven't created these hooks for key repos or key projects where everybody is benefiting from this when they're using something like Claude Code then you're missing out on some of the scaled leverage." 36:01–36:37

🛠️ Personal CLI Tools — Build Every Idea, Immediately

"It's easier to edit than author. So, let's get the authoring out of the way and then even if you completely revise the whole thing, it's a much easier starting point to work with something." 26:28 — John Lindquist

John's philosophy: the moment you have an idea, open a terminal in a new folder and brain-dump via dictation. Get something running — even if wrong — and iterate. Claire: "If you have nothing, you can't iterate on nothing." A wrong first draft is infinitely more useful than a blank page because it surfaces what you actually want. 21:05–26:09

The "Sketch" Demo

John built a CLI called "sketch" — a wrapper around Gemini's image generation CLI with pre-loaded prompt templates, colour themes, and quantity settings. Demo: user selects store type, page style, colour theme, quantity → Gemini generates N variations → pick one → drop into an AI as a wireframe reference for web development. Built in minutes; immediately useful as an ideation tool. 21:13–23:29

Why Terminal UI Beats Web UI for Personal Tools

The constrained interface forces focus. No temptation to over-design. A few questions, multi-select fields, keyboard navigation — that's it. Web UIs invite scope creep. Building CLI tools has become dramatically easier with AI — ASCII logos, interactive prompts, keyboard shortcuts — previously tedious, now trivially generatable. 24:03–25:17

📜 Analog Personal CLIs are the AI era's equivalent of personal scripts and dotfiles. Developers have always built small automation tools for themselves — what's changed is the barrier is now near-zero and the tools can be genuinely polished.

🖥️ IDE vs. Terminal — Both Required, Different Jobs

"I think you need both. I think you need an IDE and there are so many use cases for the CLIs." 42:27

The case for CLIs: Deep configuration and composability — load specific MCPs, inject specific prompts, set specific models — all in a single aliased command, runnable in the background. Claude Code's headless mode enables truly parallelisable work. 42:38–43:00

The case for IDEs: Focused editing — selecting lines, reading file context, modifying targeted sections. VS Code extension ecosystems can feed diagnostics, warnings, and errors directly back to the agent. IDE agent modes (Cursor's agent view) are increasingly streamlined for specific workflows. 43:24–43:50

🔗 Connection IDE vs. CLI maps directly to focused work vs. automated pipeline. IDE for human-in-the-loop editing; CLI for background, unattended task execution. Not competing use cases — different moments in the same workflow.

On IDE market differentiation: John argues the quality bar is so high that friction from any AI tool causes "instant dismissal." Cursor's agent mode is the right kind of bold move — seamless UX leaps, not incremental feature parity. 44:34–45:33

🔎 AI for Codebase Investigation — The Most Underused Enterprise Use Case

"Think about if I gave you infinite junior to mid-career talent who is always available, who would do the work you would do if you had unlimited amount of time and no meetings. What would you do when a ticket came in?" 49:47 — Claire Vio

The highest-ROI use of AI tools is not writing code. It's the orientation work that precedes writing code: finding who touched a file, understanding why a function is the way it is, tracing blast radius before a change. For every ticket, engineers historically spend a day or two just getting oriented — especially in legacy codebases. AI wipes out most of that. 47:00–48:47

In practice: "Find every single debug path. Find every single — why has this file changed over the course of history. Write a summary of everybody who's touched this file so I can know why this function is the way it is." 48:19–48:30

🎯 Against Consensus Skeptical engineers question AI code quality. John's reframe: even if you distrust AI for writing code, the investigation use case has almost no quality risk — the output is a summary and risk flags, not production code. This is the wedge argument for skeptical organisations. 49:03–49:26

Claire's complementary point: use AI to document what it found — so the next engineer at the same codebase has a shorter orientation cycle. "On the other end, if you're not using it to document, you're also missing out." 49:29–49:43

🔁 Prompting Reset Strategies — When to Start Over and How

"Starting over works every time." 53:45

AI drift happens when Claude is heading somewhere the developer doesn't want, and the developer keeps trying to steer it back rather than resetting. "The AI is trying to go somewhere and you want it to go over here and you keep on telling it to join your path but it still wants to get somewhere else." Once the context is polluted with misaligned attempts, corrections compound the confusion. 53:07–53:24

The Two-Move Reset

  1. Revert to the previous commit. Clean slate at the code level.
  2. Export the conversation; paste it (with relevant code files) into a different model — ChatGPT Pro, Gemini Deep Think. Get a critique from an AI not invested in the wrong direction. Diagnose where things went wrong. Revise prompt. Start fresh. 52:27–53:47
🔗 Connection The "second model as mediator" technique maps to how senior engineers use code reviews — bring in someone not invested in the current approach to see the problem fresh. Human team dynamics and AI operational dynamics have more in common than most engineers realise. 54:04–54:38

Plan mode as drift prevention: "The recent planning modes released with Claude Code and Cursor have eliminated a vast majority of that drift." Plan mode forces upfront alignment before code generation begins. Recommendation: use plan mode for anything beyond a single-file edit. 54:58–55:14

💡 Key Insight The reset is not a failure. It's faster than continuing to fight a polluted context. The sunk cost fallacy applies to AI conversations as much as to any other work.

🃏 Technique Cards

🗺️ Mermaid Diagram Context Loading

Store architecture as Mermaid markdown files; inject them all into Claude's system prompt at session startup using --append-system-prompt and a glob pattern.

Setup

Create a /memory/ai/diagrams/ directory. Generate diagram files using Claude Code (prompt: "diagram all user interactions, events, and data flows in the codebase"). Each file is a markdown file containing Mermaid diagram blocks.

Injection Command

claude --append-system-prompt "$(cat .ai/diagrams/*.md)"

Alias as cdi for one-keystroke access.

When to Generate

After a feature works; before PR merge. Retroactively on legacy codebases before starting AI-assisted development. Automatically via GitHub Actions on PR close.

Trade-offs

Cost: higher upfront token usage. Benefit: zero file reads per task, faster responses, correct dependency-impact awareness.

⚙️ Stop Hook Quality Gate

A TypeScript script using the Anthropic Agent SDK that fires on every Claude stop event, runs your type checker, feeds errors back as a blocking message, and commits clean code automatically.

Dependencies

@anthropic (Anthropic Agent SDK) · Your project's type checker / linter (bun type-check, eslint --fix, etc.)

Logic Flow

  1. Hook fires on stop event
  2. Check: did any files change? If no → exit
  3. Run bun type-check
  4. If errors → console.log(JSON.stringify({ decision: "block", reason: "TypeScript errors: " + errorOutput })) → Claude auto-corrects → loop restarts
  5. If no errors → prompt Claude to commit all changed files (exclude sensitive)

Critical Gotcha

console.log = instruction back to Claude. Use console.error for debug logs.

Configuration

Personal: settings.local.json · Team-shared: settings.json

🔤 Shell Aliases for AI Configurations

Alias your most-used Claude Code flag combinations to 1–3 letter shortcuts in your shell config.

Examples (ZSH / .zshrc)

alias x='claude --dangerously-skip-permissions'
alias h='claude --model claude-haiku-4-5-20251001'
alias cdi='claude --append-system-prompt "$(cat .ai/diagrams/*.md)"'

Project-Specific Pattern

alias cc-myproject='claude --append-system-prompt "$(cat ~/projects/myproject/.ai/diagrams/*.md)"'

When to Use Which

x (dangerous mode): trusted solo work, fast iteration. h (Haiku): quick tasks, cost-sensitive. cdi (diagrams): any session where codebase context matters.

📋 Protocol Cheat Sheet
  • Diagram generation prompt: "Walk through my codebase and generate Mermaid diagrams for all user interactions, database operations, events, and data flows"
  • Context injection alias: alias cdi='claude --append-system-prompt "$(cat .ai/diagrams/*.md)"'
  • Dangerous mode alias: alias x='claude --dangerously-skip-permissions'
  • Haiku alias: alias h='claude --model claude-haiku-4-5-20251001'
  • Stop hook return format: console.log(JSON.stringify({ decision: "block", reason: "Fix errors: " + output }))
  • Debug logs in hooks: always use console.error — never console.log
  • Team hook config: settings.json (shared) vs. settings.local.json (personal)
  • Reset sequence: revert to last commit → export conversation → paste into second model → get critique → revise prompt → start fresh
  • Plan mode trigger: anything beyond a single-file edit — activate before generating
  • Diagram timing: after feature works, before PR merge — not upfront on greenfield
  • Investigation prompt: "Write a summary of everyone who touched this file and why this function is the way it is"

⚡ The Bottom Line

  • Mermaid diagrams injected at startup eliminate codebase exploration overhead. Store architecture as markdown diagram files; inject them all via --append-system-prompt. Every session starts with full structural memory — zero file reads, faster and more accurate edits.
  • Stop hooks create a self-correcting quality pipeline. A TypeScript script using the Anthropic Agent SDK fires when Claude finishes, runs your type checker, feeds errors back as a blocking message, and auto-commits clean code. No human in the loop after task start.
  • Investigation is the highest-ROI enterprise use case — not code generation. Orienting to a legacy codebase (git history, impact tracing, risk surfacing) used to take days. It's now a 30-second prompt.
  • Plan mode has largely solved the "AI drift" problem. When Claude goes off the rails: revert to last commit, export conversation, run it through a second model as mediator, revise your prompt, start fresh. Plan mode prevents most of this from occurring.
  • Shell aliases compress your best Claude configurations to one keystroke. x for dangerous mode, h for Haiku, cdi for diagram-loaded context — set once, use forever.
